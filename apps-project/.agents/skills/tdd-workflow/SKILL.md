# TDD ワークフロー

## Red → Green → Refactor サイクル

### 1. Red：失敗するテストを書く
- 実装より先にテストを書く
- テストは「何をすべきか」を表現する
- 最初は必ず失敗させる（テストが正しく機能していることの確認）

```bash
# テスト実行（失敗を確認）
[CLAUDE.mdのテスト実行コマンド]
```

### 2. Green：テストを通す最小限の実装
- テストを通すだけの最小限のコードを書く
- 綺麗さより「動くこと」を優先
- この段階でリファクタリングしない

### 3. Refactor：コードを整理する
- テストが通ったままリファクタリング
- 重複を除去、名前を改善、構造を整理
- テストは変更しない

## テストの書き方

### 命名規則
```
[対象]_[条件]_[期待結果]
例：
- calculateTax_withZeroAmount_returnsZero
- createUser_withDuplicateEmail_throwsError
```

### 構造（AAA パターン）
```
// Arrange: 前提条件のセットアップ
// Act: テスト対象の実行
// Assert: 結果の検証
```

### テストの独立性
- 各テストは他のテストに依存しない
- テスト実行順序に関係なく同じ結果になる
- 外部状態（DB・ファイル・時刻）はモックまたはセットアップ・クリーンアップで制御

## いつTDDを使うか（CLAUDE.mdの方針に従う）

✅ 使う：
- ビジネスロジック（計算・バリデーション・変換）
- ユーティリティ関数
- バグ修正（再現するテストを先に書く）

❌ 使わない：
- UIコンポーネントの見た目
- 試作・スパイクコード
- 単純なCRUD（フレームワークのテストで十分）
