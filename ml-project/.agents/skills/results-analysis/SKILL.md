# 実験結果分析 手順

## Step 1: データ収集

対象実験の `metrics.json` を収集する。

```bash
# 全実験の結果を一覧
find outputs/ -name "metrics.json" | sort | xargs -I{} sh -c 'echo "---"; cat {}'

# 特定期間の実験
find outputs/ -name "metrics.json" -newer outputs/20240301_001_baseline/metrics.json
```

## Step 2: 比較表の作成

| 実験ID | 変更点 | val指標 | test指標 | 学習時間 | 備考 |
|--------|--------|---------|----------|----------|------|

変更点は「前回からの差分」だけ記載する（全設定を書かない）。

---

## Step 3: 分析の観点

### 3-1. 基本チェック
- [ ] ベースラインを再現できているか（±1%以内）
- [ ] train/val/test の乖離が大きくないか（過学習の兆候）
- [ ] val と test の差が大きくないか（val過学習の兆候）

### 3-2. 改善要因の特定
各変更がどれだけ寄与しているか確認する：

```
ベースライン：0.750
+ 変更A      ：0.762 (+0.012)  ← 有意な改善
+ 変更B      ：0.761 (+0.011)  ← 変更Aとほぼ同等
+ 変更A+B    ：0.775 (+0.025)  ← 相乗効果あり
```

### 3-3. 統計的な注意
- 1回の実験の差は信頼しない（少なくとも3 seeds で確認する）
- 改善幅がノイズレベルかどうか判断する（目安：±0.005以下は誤差範囲）
- データセットが小さい場合は特に注意する

### 3-4. 失敗実験の分析
改善しなかった実験から学ぶ：
- 仮説は正しかったか
- 実装に問題があったか
- 条件設定が悪かったか（learning rateが高すぎるなど）

---

## Step 4: 可視化のポイント

学習曲線で確認すること：
- val lossが下がり続けているか（まだ改善余地があるか）
- val lossが上がり始めている点（early stopping のタイミング）
- train と val の乖離（過学習の程度）

---

## Step 5: 次のアクション提言

以下のフレームワークで優先順位をつける：

**優先度高（すぐ試す）**：
- 小さい変更で大きな改善が期待できるもの
- 失敗の原因が明確で修正方法が分かっているもの

**優先度中（次のイテレーションで）**：
- 有望だが実装コストが高いもの
- 複数の仮説のどれが正しいか検証が必要なもの

**優先度低（後で）**：
- 改善幅が小さそうなもの
- 前提条件が揃っていないもの

---

## よくある落とし穴

- テストデータで最適化しない（testは最後の1回だけ使う）
- 改善が小さいのに「効果あり」と結論づけない
- 一度に複数の変更を加えて何が効いたか分からなくなる
- negative result を記録しない（同じ失敗を繰り返す原因になる）
